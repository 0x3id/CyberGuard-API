01 — Platform Overview

What Problem Does CyberGuard Solve?

Professional penetration testing today is fragmented. Pentesters juggle Burp Suite, Nmap, custom scripts, spreadsheets for findings, and email chains for client communication — all disconnected. CyberGuard replaces this with a single, structured, multi-user platform covering the entire security assessment lifecycle from project creation to final PDF delivery.

Problem 1
Scattered Tooling
Pentesters switch between multiple disconnected tools. Findings are written in Notepad, evidence lives in a screenshots folder, and reports are assembled manually in Word — taking hours per client.
Problem 2
No Real Collaboration
Multiple pentesters on the same engagement share findings via email or Slack. There is no single source of truth, no role enforcement, and no record of who changed what and when.
Problem 3
Manual Reporting
Every report is rebuilt from scratch. Copying CVSS scores, formatting findings, and writing executive summaries consumes time that should be spent on actual security work.
✅
CyberGuard's Solution
One unified SaaS workspace: collaborative project management, automated scanning via isolated Docker containers, structured finding management with CVSS scoring, phishing simulation, and instant PDF report generation — all accessible to every team member according to their role.
02 — Complete User Journey

From Registration to Delivered Report

The following flow describes the full lifecycle of a penetration testing engagement on CyberGuard, covering every major decision point and all possible branches.

01
Account Registration & Subscription
A new user registers with their email address and a secure password. The system sends a verification email — the account is inactive until confirmed. Upon first login, the user selects a subscription plan that enforces resource limits (max projects, targets, scans per month). Two-Factor Authentication (2FA via TOTP) is available and strongly recommended.
Path A
Individual Freelancer
Registers as a personal account. Works on personal projects with no organizational structure. Selects Free or Pro plan. Can invite collaborators directly to individual projects without creating an organization.
Path B
Security Team Lead
Registers personally, then creates an Organization. Invites team members with roles (Admin / Member / Viewer). All organization members share a subscription quota. Projects can be owned by the org itself.
Path C
Corporate Client
Registers as Enterprise. Primarily uses the platform for internal security assessments and phishing awareness campaigns. May invite an external pentesting firm as an Editor on their projects.

02
Project Creation & Ownership Model
The user creates a new Project and assigns it to an owner — either themselves (personal) or an Organization they belong to. The project holds a name, description, start/end dates, and a maximum collaborator limit enforced by their subscription tier. Projects support Soft Delete — archived projects are preserved in the database and can be restored.
ℹ️
Polymorphic Ownership Design
The projects table uses a polymorphic relationship via owner_type (user or organization) and owner_id. This allows a single project model to be owned by either entity without a separate table per type — a clean, extensible architecture.
Personal Ownership
owner_type = "user" — the creating user is both the owner and the first collaborator with role owner. Only this user can archive or delete the project, and only they can remove other collaborators.
Organization Ownership
owner_type = "organization" — the project belongs to the org. Any org Admin can manage the project. The user who created it is tracked via created_by for audit purposes.

03
Team Collaboration & Invite System
The Project Owner invites additional pentesters as collaborators. Two invitation methods exist: a shareable link with an embedded token, or a direct email invite. Each invitation is stored in project_invitations with a unique token and a 7-day expiry. Once accepted, the user is added to project_collaborators and gains immediate access based on their assigned role.
⚠️
Critical Business Rule — Owner Self-Removal Prevention
The project Owner cannot remove themselves from a project. This prevents accidental orphaning. To transfer ownership, the Owner must first promote another member to Owner, then remove themselves. Role changes take effect immediately — no session restart required.

👑 Owner
Opens project → clicks Invite Member → selects role (Editor or Viewer) → chooses invite method (link or email)
02
⚙️ System
Generates a unique token → creates record in project_invitations with status pending and a 7-day expiry timestamp

04
👤 Invitee
Opens the link → sees project name, assigned role, and inviter's identity → clicks Accept Invitation

Edge Case: Token Expired
If the invitee opens the link after 7 days, the system returns 422 Unprocessable Entity. The Owner must generate a fresh invitation — expired tokens cannot be reused or extended.
Edge Case: Already a Member
If the invitee is already a collaborator on the project, the system rejects the acceptance with an appropriate error rather than creating a duplicate record in project_collaborators.
Edge Case: Owner Removes Member
The Owner can remove any Editor or Viewer at any time. The project_collaborators row is soft-deleted. The removed member immediately loses all API access — any active session is rejected on the next call.

04
Adding & Verifying Targets
Before scanning, the pentester defines one or more Targets inside the project. A target is the asset being tested — it can be a domain name, an IP address, or an IP network range. The system validates the format and optionally verifies domain ownership (DNS TXT record challenge) to prevent unauthorized testing against assets the client does not own. Each target maintains an independent risk_score recalculated after every completed scan.
Target Type: Domain
e.g. techcorp.com. Supports all web-layer scan modules: SQLi, XSS, CSRF, open redirect, subdomain enumeration, SSL/TLS analysis, HTTP security header inspection, CORS misconfiguration.
Target Type: IP Address
e.g. 192.168.1.10. Supports network-layer modules: port scanning, service banner grabbing, WHOIS lookup, and service fingerprinting via Nmap-compatible tooling.
Target Type: Network Range
e.g. 192.168.1.0/24. Runs a host discovery sweep (ping sweep) followed by port and service enumeration across all live hosts discovered in the CIDR range.

05
Scan Execution — Auto & Targeted
The pentester launches a scan against a target. In Auto Scan mode, all active modules run sequentially. In Targeted Scan mode, the user selects specific modules. Either way, a scan_jobs record is created with status pending, dispatched to the Laravel Queue via Redis. A worker picks up the job, spins up an isolated Docker container per module, executes the tool, parses output, and persists findings automatically.
🐳
Why Docker Containers?
Each scan module runs inside its own Docker container. This provides complete process isolation (a crash in one module does not affect others), prevents resource exhaustion on the host, and allows different tool versions to coexist. Containers are destroyed immediately after the scan completes — no persistent state between runs.

Scan Lifecycle:

01
👤 User
Selects target → chooses Auto or Targeted scan → selects modules (if targeted) → clicks Launch → frontend sends POST /api/scans
02
⚙️ Laravel
Validates quota (scans remaining this month per subscription), creates scan_jobs record with status = pending, dispatches job to Redis queue
03
🐳 Worker
Queue worker picks up job → updates status = running → spins up Docker container for each selected module in isolation
04
🐳 Module
Tool executes against target, generates structured output → container exits, worker parses results, creates findings records with severity and full CVSS data
05
⚙️ Laravel
Updates status = completed (or failed on error) → recalculates targets.risk_score → broadcasts real-time event via WebSocket to all project members
06
📊 Frontend
Receives WebSocket event → updates scan progress bar live → new findings appear in the list without page reload — visible to all connected collaborators simultaneously
Failure Scenario
If the Docker container exits with a non-zero code (target unreachable, tool crash, timeout), the job is marked failed. The specific error is persisted in scan_jobs.error_message. The user sees the error in the UI and can retry.
Quota Exceeded
If the user has consumed all their monthly scans (enforced via user_subscriptions.max_scans_per_month), the API returns 403 Forbidden with a plan upgrade prompt. The scan is not queued.

06
Finding Management & Evidence Collection
Every vulnerability discovered by a scan (or manually added by a pentester) is stored as a finding record. Findings are classified by severity (Critical → Info) and assigned a full CVSS v3 score and vector string. Where applicable, a CVE identifier is linked for reference. Each finding tracks its remediation status through a defined lifecycle, and supports multiple evidence attachments.
Status: open
Newly discovered, no remediation action taken. Contributes fully to the target's risk score calculation. Visible to all collaborators with at least Viewer access.
Status: in_progress
The development or operations team has acknowledged the finding and is actively working on a fix. Partially discounted from risk score to reflect ongoing remediation effort.
Status: resolved
A fix has been deployed and the pentester has re-tested and confirmed remediation. No longer counts toward risk score. Historical record is permanently preserved for audit trail.
Status: accepted_risk
The client has formally acknowledged the vulnerability and accepted the risk without fixing it (e.g. legacy system, cost/benefit decision). Documented with a written rationale — excluded from risk score but visible in reports.
📸
Evidence Attachments
Each finding supports multiple evidence files stored in finding_evidences. These include screenshots (auto-captured via Puppeteer/Browsershot), raw HTTP request/response pairs captured during the scan, tool output logs, and any file manually uploaded by a team member. All files are stored in Amazon S3 or MinIO (for self-hosted deployments).

07
Phishing Simulation (Optional Module)
When the scope includes human-layer testing, the pentester runs a phishing simulation campaign against the client's employees. This module tests security awareness by sending simulated phishing emails and tracking interaction at the individual level using unique per-employee tokens for granular attribution.

Campaign Lifecycle:

01
✏️ Pentester
Creates campaign: writes email subject, body, sender name; sets phishing URL (must be a pre-verified domain within the engagement scope)
02
✏️ Pentester
Uploads employee CSV (name, email, department) → system creates one phishing_targets record per employee, each with a unique, non-guessable tracking token
03
⚙️ System
Campaign marked running → queued email jobs dispatched via Laravel Mail at controlled intervals to avoid triggering spam filters
04
📧 Tracking
Each email contains a 1×1 pixel linked to GET /track/{token}/open — when loaded by the email client, records an opened event in phishing_events
05
👤 Employee
If employee clicks the phishing link → clicked event recorded with timestamp and IP. If they complete the fake form → submitted event recorded — credentials never stored
06
📊 Analysis
Awareness score (0–100) calculated per employee based on all events. Department-level aggregation identifies high-risk teams. Summary visible in real-time dashboard.
Best Case
Employee does not open the email, or opens it but does not click the link. Score: 80–100. Recommended action: positive reinforcement, basic awareness refresher.
Medium Risk
Employee opened and clicked the link but did not submit credentials. Score: 40–60. Recommended: targeted phishing awareness training session.
High Risk
Employee opened, clicked, and submitted credentials into the fake form. Score: 0–20. Recommended: mandatory security training, account security review, immediate MFA enforcement.

08
PDF Report Generation & Delivery
When the engagement is complete, the pentester generates a professional PDF report with a single action. The report is compiled by DomPDF from a server-rendered Blade template that aggregates all project data — findings, CVSS scores, evidence, risk scores, and phishing results. The generated file is stored in S3 and linked as a project_reports record for permanent access.
📄
Report Contents
Executive summary · Project scope and methodology · All findings grouped by severity · CVSS v3 score and vector per finding · Remediation recommendations per finding · Evidence screenshots inline · Historical risk score trend chart · Phishing awareness results (if applicable) · Generated timestamp and pentester attribution
Per-Project Report
Covers all targets and findings within a single project. Suitable for the final client deliverable after a complete engagement. Includes cross-target risk comparison.
Per-Target Report
Scoped to a single target. Useful for partial deliverables, milestone reviews, or when different assets belong to different stakeholders within the same organization.

03 — Access Control

Role-Based Permission System

CyberGuard enforces a three-tier role system at the project level. Permissions are checked server-side on every API endpoint via Laravel Policy classes — the frontend reflects the same state, but the backend is the single source of truth. Roles are stored in project_collaborators.role.

Role
Project & Targets
Scans & Findings
Members & Reports
👑 Owner
Create, edit, archive, delete. Manage all project settings and targets. Transfer ownership.
Launch any scan type. Create, update, and delete findings. Upload and delete evidence.
✓ Invite, remove, change roles of any member. Generate and delete PDF reports.
✏️ Editor
Edit project details and add/remove targets. Cannot archive or delete the project itself.
Launch scans. Create and update findings and their status. Upload evidence screenshots.
✗ Cannot invite or remove members. Can generate but not delete PDF reports.
👁 Viewer
Read-only. Cannot modify project, targets, or any settings whatsoever.
Can view findings and evidence. Cannot launch scans or modify finding status.
✗ No management access. Can download (not generate) existing reports.
⚠️
Owner Self-Removal Prevention
The system explicitly checks whether the user attempting a leave or remove action is the sole Owner. If so, the request is rejected with 403 Forbidden. To leave a project they own, the Owner must first promote another collaborator to Owner, then remove themselves. This rule is enforced at the policy level — it cannot be bypassed via the API.

06 — Project Summary

CyberGuard at a Glance

What is CyberGuard?
CyberGuard is a multi-tenant SaaS platform purpose-built for professional penetration testing and security assessment. It consolidates every stage of a security engagement — from project creation and team collaboration through automated vulnerability scanning, structured finding management, phishing simulation, and professional report delivery — into a single, cohesive product. Whether used by a solo freelancer, a corporate security team, or an enterprise client running internal assessments, CyberGuard scales to the workflow without friction.
🎯
Project & Target Management
Flexible polymorphic ownership — projects belong to a user or an organization. Targets support domain, IP, and CIDR range types. All quotas enforced by subscription plan.
🔍
Automated Scanning
12 modules dispatched via Laravel Queue to isolated Docker containers. Auto Scan runs all modules; Targeted Scan lets the user select. Real-time WebSocket progress updates.
🐛
Finding Management
Full CVSS v3 scoring, CVE linking, 4-state status lifecycle, multi-file S3 evidence attachments, and per-finding remediation guidance.
👥
Team Collaboration
Owner/Editor/Viewer RBAC enforced server-side via Laravel Policies. Invite by link or email with 7-day token expiry. Full audit trail on every action.
🎣
Phishing Simulation
End-to-end campaign management with per-employee event tracking (sent → opened → clicked → submitted) and department-level awareness scoring.
📊
Reports & Risk Score
One-click DomPDF generation with executive summary, all findings, CVSS data, inline evidence, and historical risk score trend — stored permanently in S3.
